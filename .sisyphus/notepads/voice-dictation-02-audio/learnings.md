# Audio Recorder Implementation Learnings

## CPAL Usage Patterns
- Use `cpal::default_host()` to get the default audio host
- Use `host.default_input_device()` to get the default microphone
- Device configurations should be obtained from the device itself rather than created manually
- Handle different sample formats (F32, I16, U16) in the input stream callback
- Convert multi-channel audio to mono by taking the first channel
- Use crossbeam channels for real-time audio streaming between threads

## Error Handling
- Wrap cpal errors in AppError::AudioError for consistent error handling
- Validate sample format support but be flexible with sample rate and channels
- Handle device enumeration failures gracefully

## Testing Strategy
- Unit tests can mock audio processing without actual hardware
- Test sample format conversion logic independently
- Duration tracking can be tested by manipulating Instant values

## PipeWire Backend Support
- CPAL automatically handles PipeWire on Pop!_OS and other Linux distributions
- No special configuration needed for PipeWire - it's detected as the default host
- The same code works across Windows, macOS, and Linux with different backends

## Performance Considerations
- Use bounded channels to prevent memory buildup during recording
- Convert samples to f32 immediately in the callback to avoid additional processing later
- Implement 10-minute max duration to prevent unbounded memory usage

# Audio Encoding Module Learnings

## Hound Crate Usage
- Version 3.5.1 is the latest stable version available on crates.io
- `WavWriter::new()` takes a writer (like `Cursor<Vec<u8>>`) and `WavSpec`
- **Important**: In hound 3.5.1, `WavWriter` does NOT have `into_inner()` or `get_ref()` methods
- Workaround: Use `Rc<RefCell<Vec<u8>>>` to share the buffer between cursor and main function
- Always call `writer.finalize()` to write the final WAV header before accessing data
- `finalize()` returns `Result<(), Error>` and must be called to complete the WAV file

## Sample Format Handling
- F32 samples can be written directly to hound when using Float sample format
- I16 samples require conversion from f32: clamp to [-1.0, 1.0] and scale to i16 range
- WAV header is automatically generated by hound based on `WavSpec`

## Memory Efficiency
- Using `Rc<RefCell<Vec<u8>>>` with `Cursor` allows in-memory encoding without temporary files
- This approach is memory-efficient and suitable for the 25MB Groq API limit
- Avoids file system operations entirely

## Size Validation
- 25MB = 25 * 1024 * 1024 bytes
- For 16kHz mono 16-bit audio: ~32KB per second (~19.2MB for 10 minutes)
- For 16kHz mono 32-bit float audio: ~64KB per second (~38.4MB for 10 minutes)
- Size validation must happen after encoding to get accurate byte count

## Error Handling
- All hound errors are converted to `AppError::AudioError` with descriptive messages
- Size limit violations return specific error messages mentioning Groq API limits
- Finalize errors are captured and converted to appropriate error types

## Tauri Commands Implementation Learnings

### Task 4: Tauri Commands for Audio
- **Challenge**: `AudioRecorder` contains `cpal::Stream` which is not `Send`, preventing direct storage in Tauri state
- **Solution**: Use global static `Lazy<Arc<Mutex<Option<AudioRecorder>>>>` for recorder state, while Tauri state only stores `AudioConfig`
- **Commands implemented**:
  - `start_audio_recording`: Creates recorder, starts recording, emits `audio-recording-started`
  - `stop_audio_recording`: Stops recording, encodes to WAV, emits `audio-recording-stopped` and `audio-data-ready`, returns WAV bytes
- **Event emission**: Used `app.emit()` with JSON payloads for frontend communication
- **Thread safety**: Global mutex ensures safe access to recorder across async command calls
- **State management**: Tauri's `manage()` used to store `AudioState` with default configuration
- **Compilation**: Code compiles successfully with `cargo check --lib`
- **Dependencies**: Leveraged existing `once_cell` dependency for global static initialization

### Key Insights:
- CPAL's non-`Send` nature requires careful state management in multi-threaded environments
- Tauri's state system requires `Send + Sync + 'static` types, limiting direct storage of audio streams
- Global static variables with proper synchronization provide a viable workaround
- Event-driven architecture allows frontend to react to recording state changes
- Default audio configuration (16kHz, mono, F32) aligns with Groq Whisper API requirements
